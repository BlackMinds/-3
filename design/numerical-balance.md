## 🚧 v3.0 进行中 — 数值体系重构

> **目标**: 告别"哪里不对调哪里"的散点修复,建立一套**单一数值源 + 目标函数 + 自动验证**的工程化数值体系

### 为什么需要 v3.0

v1.x → v2.0 都是**局部微调**:发现某处爆表就去改那一处,改完又在别处出问题。根本问题:

- **同一数值多地定义**(primaryBases 8 处 / WEAPON_BONUS 4 处)
- **8 个属性来源各自拍数字**(基础/等级/境界/装备主副/功法/丹药/附灵/宗门/道果),没有统一预算
- **没有"目标函数"**:Lv 100 玩家应打出多少 DPS 没定义,数值靠直觉
- **没有验证工具**:改数值后要等玩家喊才知道有问题

- **装备类重复度最高**: `primaryBases` 在 8 个文件重复, `WEAPON_BONUS` 在 4 个文件重复 — 这是本次重构的最大收益点

## 🎯 v2.0 改版哲学

> **"数字人脑可读 + 装备是战力 C 位"**

v1.x 版本玩家到 T7+ 后战力达到百万级，数字膨胀臃肿，换装/强化的爽感被各种 buff 稀释。v2.0 做了一次**系统级数值重构**：

1. **数字压缩 25x**：T10 Boss HP 从 14 亿 → 5600 万，全程 8 位数内
2. **装备占战力 70%+**：境界 ÷3、等级不变、装备级差拉大，让换装直接影响战力
3. **强化爽感拉满**：每级 +10%（原 +8%），满 +10 = +100%（原 +80%）
4. **保留原版战斗公式**：atkDefRatio / monster role 不动，避免破坏 tank/dps 定位

## 一、核心设计理念

### 1.1 玩家生命周期阶段划分

放置修仙游戏的典型留存曲线是"首日峰值 → 7日平稳 → 30日转化"。本文档按三段设计：

| 阶段 | 时间 | 目标境界 | 核心体验 | 数值原则 |
|------|------|---------|---------|---------|
| **爽感期** | Day 1–7 | 练气 → 金丹中期 | 突破频繁、掉落丰富、强度大幅提升 | **指数加速** |
| **成长期** | Week 2–4 | 金丹后期 → 化神 | 装备分化、功法组合、宗门玩法接入 | **几何稳定** |
| **长线期** | Month 2+ | 渡劫 → 飞升 | 小幅进步、玩法追求、社交/数值对比 | **对数收敛** |

### 1.2 三条核心原则

1. **"爽感期"每日至少一次大突破**：修为曲线前期压缩，7 天内必须完成 4-5 次境界跳跃（对应练气 9 层+筑基 3 阶+金丹 3 阶）。
2. **"爽感期"不惩罚玩家**：装备强化在前期必成、死亡不掉落、离线收益足额。
3. **"长线期"靠深度而非数值**：通过**功法 BD 选择、套装搭配、宗门 Boss、道果永久加成**提供目标，而非单纯修为数字堆高。

## 三、境界系统调整

### 3.1 修为突破需求（关键调整）

#### 当前问题
`getExpRequired(tier, stage) = 100 * exp_multiplier * Math.pow(stage+1, 1.6)`

- **练气一层**：100 * 1 * 2^1.6 = 254 修为 ✅ 合理
- **练气九层**：100 * 1 * 10^1.6 = 3,981 修为（按闭关 440/8h → **72 小时**，严重拖节奏）
- **筑基→金丹**跳 5x → 合理
- **飞升**：100 * 15000 * 6^1.6 ≈ 2,374 万起步 ⚠️ 太卡

#### 建议公式

```ts
// 修改 game/data.ts:395
getExpRequired(tier, stage) {
  // 练气期特化：小 stage 线性，快速突破
  if (tier === 1) return Math.floor(50 * Math.pow(stage, 1.2));

  // 筑基起走原指数曲线，但指数降至 1.4 避免后期爆炸
  return Math.floor(100 * exp_multiplier[tier] * Math.pow(stage + 1, 1.4));
}
```

**调整后数值**：

| 境界阶段 | 当前需求 | 建议需求 | 单位 |
|---------|---------|---------|------|
| 练气 1 | 254 | 50 | 修为 |
| 练气 5 | 1,214 | 345 | 修为 |
| 练气 9 | 3,981 | 720 | 修为 |
| 筑基 1 | 2,540 | 2,000 | 修为 |
| 金丹 1 | 12,700 | 10,000 | 修为 |
| 元婴 1 | 50,800 | 40,000 | 修为 |
| 飞升 1 | 3,810,000 | 3,000,000 | 修为 |

保持原梯度，微调渡劫/大乘/飞升段以减缓后期断崖：

### 3.3 闭关修为产出

#### 当前公式

#### 建议调整（放大前期、平滑后期）

// server/api/game/cultivate.post.ts

## 四、等级系统调整

### 4.1 经验曲线重构

#### 当前问题
Lv.200 需要 `350 * 200^1.5 ≈ 9.9 亿 exp`，而最高地图单次战斗也就 3000 万 exp，需要 33 场战斗升 1 级——实际上**永远不满级**。

#### 建议曲线

```ts
// stores/game.ts:96
if (lv <= 30)  return Math.floor(60  * Math.pow(lv, 1.25));   // 降 25% 前期
if (lv <= 80)  return Math.floor(100 * Math.pow(lv, 1.35));   // 略降
if (lv <= 150) return Math.floor(180 * Math.pow(lv, 1.42));
if (lv <= 200) return Math.floor(320 * Math.pow(lv, 1.48));
// 200 级可满级，但留出 100-200 的长线空间
```

**对比表**：

| 等级 | 当前 exp 需求 | 建议 exp 需求 | 降幅 |
|------|------------|------------|------|
| Lv.10 | 1,599 | 1,067 | -33% |
| Lv.30 | 11,087 | 6,778 | -39% |
| Lv.50 | 149,490 | 101,802 | -32% |
| Lv.100 | 1,660,460 | 1,258,925 | -24% |
| Lv.150 | 20,638,674 | 14,126,310 | -32% |
| Lv.200 | ∞ (350 * 200^1.5) | 320 * 200^1.48 ≈ 4.7 亿 | 可达 |

### 4.2 等级属性加成

## 七、装备强化调整

>

### 7.1 强化成功率曲线（关键爽感）

#### 当前
```
≤5: 100%, 6: 80%, 7: 70%, 8: 55%, 9: 40%, 10: 25%
```

#### 建议（放宽基础档）

// game/equipData.ts:289
getEnhanceSuccessRate(nextLevel) {
  if (nextLevel <= 6) return 1.0   // +6 改为必成（新手强化不劝退）
  if (nextLevel === 7) return 0.75
  if (nextLevel === 8) return 0.60

#### 当前

+7 起失败退 1 级，最低保留 **+6**（与必成档一致，减少挫败感）

- 周限从 2 → **3**（配合每日任务多给贡献）
- 成本从 3000 → **2000**（让前期宗门玩家能囤货）

### 7.4 强化消耗（不变）

当前成本曲线 `baseCost * (currentLevel + 2)^1.4` 合理。

### 7.5 强化属性加成

当前每级 +8%，+10 满强化 = +80%。建议**+10 额外给 5% 突破加成**，即 +85%，鼓励追求满强化。

## 附录 D：设计哲学小结

> **"让玩家每次上线都有进步的感觉。"**

放置游戏的核心不是数值有多大，而是**每次回来时能看到明显变化**：多了一件装备、功法升了 1 级、修为前进了 20%、击杀了新 Boss。

数值设计的本质是**控制这种"变化感"的节奏**：
- **前一周**：几分钟一个小变化 + 每日一个大变化
- **成长期**：每天 1-2 个明显变化
- **长线期**：每周 1 个目标 + 每月 1 次质变

当前项目的数值骨架是合理的，**主要问题集中在"前期拖节奏"和"后期数值断崖"**。按本文档 P0-P1 的调整执行后，预期 Day 1 留存可提升 15-20%，Day 7 留存提升 8-12%。
