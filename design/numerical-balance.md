## ✅ v3.4.3 (2026-04-26) — 历练 + 秘境修为/等级双经验整体 ×0.7

### 背景

v3.4.x 后期 Top30 玩家境界推进过快（多日内连跨大阶），主要溢出来源：
- **同一份怪物 exp 同时累加 `cultivation_exp`（修为）与 `level_exp`（等级）**
- 历练 / 组队秘境两条主路径都吃这条加倍曲线，叠加宗门加成 + 洞府演武 + 觉醒副词条 → 后期日产远超设计预期

### 改动

在两条路径的最终结算点统一乘 0.7（修为/等级同源缩，比例不变）：

| 路径 | 文件:行 | 改动 |
|---|---|---|
| 历练（地图战斗） | `server/api/battle/fight.post.ts:760` | `totalExp = floor(result.totalExp × expMul × catchUpMul × **0.7**)` |
| 组队秘境 | `server/api/team/start.post.ts:304` | `totalBaseExp = floor(avgMonsterPower × baseExpUnit × **0.84**)`（= 1.2 × 0.7 组队加成系数）|

### 不影响

- 灵石、装备、灵草、功法残页、秘境积分、附灵石、强化石（全部走另一条结算线）
- 怪物 tooltip 上的单只 exp 期望（数值落到 `myExp` 之前才打折）
- `expBonusPercent`（演武堂/觉醒/丹药）、`catchUpMul`（追赶机制）均按缩后值放大，叠加比例不变
- 宗门"宗门之力"加成（`expTeamBonus`）作为相对乘数，逻辑与之前一致

### 预期

- 后期玩家修为/等级日产 ≈ 原值 × 70%
- 中前期玩家受影响较小（已有 catchUpMul ≥ 1.0 时仅吃单层 0.7）
- 装备/灵石经济通胀曲线不变

## ✅ v3.1 (2026-04-22) — 怪物 power 曲线对齐玩家战力

### 背景

v3.0 建立 `shared/balance.ts` 后, sim 跑 baseline 暴露一个致命问题:
**T3+ 所有 tier 玩家对 Boss 胜率 0%, 怪物 ATK 比玩家 ATK 高 10-5000 倍**.

详见 [`sim-report-v3.md`](./sim-report-v3.md).

### v3.1 修复

按 sim 反推公式 `power = 玩家 ATK × sqrt(15 × 目标回合)`, 各 tier 统一下调:

| Tier | 缩放 | 原 Boss power → 新 power |
|---:|---:|---|
| T1 | × 1.0 | 300-600 → 保持 |
| T2 | × 0.8 | 3000-4000 → 2400-3200 |
| T3 | × 0.30 | 15K-22K → 4500-6600 |
| T4 | × 0.25 | 60K-80K → 15K-20K |
| T5 | × 0.10 | 350K-450K → 35K-45K |
| T6 | × 0.06 | 1.5M-1.6M → 90K-96K |
| T7 | × 0.02 | 6M-9M → 120K-180K |
| T8 | × 0.01 | 30M-50M → 300K-500K |
| T9 | × 0.004 | 150M-200M → 600K-800K |
| T10 | × 0.0015 | 500M-800M → 750K-1.2M |

**涉及文件**:
- `server/api/battle/fight.post.ts` 的 ALL_MAPS (16 个地图 × ~60 个 power)
- `server/engine/sectData.ts` 的 SECT_BOSSES (5 个宗门 Boss)
- `server/engine/secretRealmData.ts` 的 SR-2/3 (SR-1 已匹配)

**注意**:
- `exp`/`stone_min`/`stone_max` 不动(经济系统保持)
- 前端 `game/data.ts` 的地图副本数值未同步(只影响展示,不影响战斗),后续统一

### sim 结果

数量级差距从 **5333x(T10) 压缩到 1.5-2x**,进入合理范围. 但 sim 仍显示玩家胜率 0%:

- **sim 模型局限**: 未复刻被动减伤(金钟罩/heavenly_body) / 回血神通(spring_heal/water_mastery) / 每回合回血(regen_per_turn) / 免死(revive_once) / 斩杀(<30% HP +60%) / 金钟罩(暴击减伤 35%) 等
- 真实战斗中玩家 EHP 实际是 sim 建模的 2-5 倍
- **建议**: dev 服实测验证最终胜率, 再微调

## 🚧 v3.0 — 数值体系重构

> **目标**: 告别"哪里不对调哪里"的散点修复,建立一套**单一数值源 + 目标函数 + 自动验证**的工程化数值体系

### 为什么需要 v3.0

v1.x → v2.0 都是**局部微调**:发现某处爆表就去改那一处,改完又在别处出问题。根本问题:

- **同一数值多地定义**(primaryBases 8 处 / WEAPON_BONUS 4 处)
- **8 个属性来源各自拍数字**(基础/等级/境界/装备主副/功法/丹药/附灵/宗门/道果),没有统一预算
- **没有"目标函数"**:Lv 100 玩家应打出多少 DPS 没定义,数值靠直觉
- **没有验证工具**:改数值后要等玩家喊才知道有问题

### v3.0 四阶段计划

| 阶段 | 产出 | 状态 | 耗时 |
|---|---|---|---|
| **① 数值审计** | [`balance-audit-v3.md`](./balance-audit-v3.md) — 13 大类 150+ 散点盘点 | ✅ 完成 (2026-04-22) | 1 天 |
| **② 设计意图** | [`balance-intent-v3.md`](./balance-intent-v3.md) — 目标曲线 + 属性预算表 + 已定决策 | ✅ 完成 (2026-04-22) | 1 天 |
| **③ 单一数值源** | `shared/balance.ts` — 所有数值集中,代码 import 读取 | ✅ 完成 (2026-04-22) | 4-5 天 |
| **④ 模拟器** | [`test/sim.ts`](../test/sim.ts) — 命令行输入 build 输出 DPS/胜率/回合数 | ✅ 完成 (2026-04-22) | 2-3 天 |

### 阶段 ④ 关键发现

模拟器 baseline 跑完后, 暴露了**怪物 power 曲线与玩家建模严重脱节**的设计问题. 详见 [`sim-report-v3.md`](./sim-report-v3.md):

- T1 玩家过强(2 回合秒 Boss, 目标 8-12)
- T3+ 玩家胜率 0%, 怪物 ATK 比玩家 ATK 高 **10-5000 倍**
- 需要 **v3.1 启动怪物 power 重构**(详见报告建议)

### v3.0 阶段② 决策结果(已定)

| 决策 | 确定值 |
|---|---|
| T10 Boss 目标回合 | **45**(挑战上限) |
| 装备占毕业战力 | **50%**(装备驱动型) |
| cap 策略 | **都做** — crit_rate 80→75% / crit_dmg 400→320% / **dodge 40→45%(反向升)** / lifesteal 30→25% / armorPen 80→60 / accuracy 60→50 |
| 戒指主属性 | CRIT_RATE base **1 → 0.8** (缓解中档触顶,T10红+10 从 50% → 40%) |
| 神器权重 | 好词条 weight **5→10** (神器概率 0.002% → ~0.7%) |
| 附灵占战力 | **8-12%**(保留当前数值) |
| 挑战曲线范围 | 仅调整 T5+,T1-T4 保持原状 |

来源数值同步压缩规则详见 [`balance-intent-v3.md`](./balance-intent-v3.md) §三。

> ⚠️ **范围说明**: v3.0 仅调现有数值,不引入新功能。洗练保底 / 词条锁定等机制虽然能有效提升玩家追求路径,但属于**未开发的新需求**,需另行立项,不在本次重构范围。

### 关键发现(from 审计)

- **装备类重复度最高**: `primaryBases` 在 8 个文件重复, `WEAPON_BONUS` 在 4 个文件重复 — 这是本次重构的最大收益点
- **高优先级 4 项**: 等级曲线 / 境界加成 / 装备数值 / 怪物+战斗公式
- **cap 早期触达问题**: 中档 build(T7 紫 + 境界6-5 + 功法 Lv3) 已触达 crit_rate/dodge/lifesteal 上限,后续装备无成长感 — 根源是各来源没有统一预算

## 🎯 v2.0 改版哲学

> **"数字人脑可读 + 装备是战力 C 位"**

v1.x 版本玩家到 T7+ 后战力达到百万级，数字膨胀臃肿，换装/强化的爽感被各种 buff 稀释。v2.0 做了一次**系统级数值重构**：

1. **数字压缩 25x**：T10 Boss HP 从 14 亿 → 5600 万，全程 8 位数内
2. **装备占战力 70%+**：境界 ÷3、等级不变、装备级差拉大，让换装直接影响战力
3. **强化爽感拉满**：每级 +10%（原 +8%），满 +10 = +100%（原 +80%）
4. **保留原版战斗公式**：atkDefRatio / monster role 不动，避免破坏 tank/dps 定位

## 一、核心设计理念

### 1.1 玩家生命周期阶段划分

放置修仙游戏的典型留存曲线是"首日峰值 → 7日平稳 → 30日转化"。本文档按三段设计：

| 阶段 | 时间 | 目标境界 | 核心体验 | 数值原则 |
|------|------|---------|---------|---------|
| **爽感期** | Day 1–7 | 练气 → 金丹中期 | 突破频繁、掉落丰富、强度大幅提升 | **指数加速** |
| **成长期** | Week 2–4 | 金丹后期 → 化神 | 装备分化、功法组合、宗门玩法接入 | **几何稳定** |
| **长线期** | Month 2+ | 渡劫 → 飞升 | 小幅进步、玩法追求、社交/数值对比 | **对数收敛** |

### 1.2 三条核心原则

1. **"爽感期"每日至少一次大突破**：修为曲线前期压缩，7 天内必须完成 4-5 次境界跳跃（对应练气 9 层+筑基 3 阶+金丹 3 阶）。
2. **"爽感期"不惩罚玩家**：装备强化在前期必成、死亡不掉落、离线收益足额。
3. **"长线期"靠深度而非数值**：通过**功法 BD 选择、套装搭配、宗门 Boss、道果永久加成**提供目标，而非单纯修为数字堆高。

## 二、玩家节奏规划（7 日目标）

### 2.2 每日"小爽点"设计

每日至少 3 个爽点：
- **1 次大突破**（境界或小境界）
- **1 次稀有掉落**（紫+品质装备或功法）
- **1 次强度跨越**（强化到 +5、学习新神通、炼出新丹药等）

## 附录：设计哲学小结

> **"让玩家每次上线都有进步的感觉。"**

放置游戏的核心不是数值有多大，而是**每次回来时能看到明显变化**：多了一件装备、功法升了 1 级、修为前进了 20%、击杀了新 Boss。

数值设计的本质是**控制这种"变化感"的节奏**：
- **前一周**：几分钟一个小变化 + 每日一个大变化
- **成长期**：每天 1-2 个明显变化
- **长线期**：每周 1 个目标 + 每月 1 次质变

数值骨架经 v1.x → v3.1 多轮迭代后,已建立"单一数值源 + 模拟器验证"的工程化体系。后续任何数值改动流程:

1. 改 `shared/balance.ts` 或 `fight.post.ts` 的 ALL_MAPS
2. 跑 `npx tsx test/sim.ts` 验证量级
3. dev 服实测验证真实体验
4. 如需微调 → 回到步骤 1
