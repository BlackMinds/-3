# 通天塔系统 — 详细设计文档

> 版本：v1.0 | 日期：2026-05-06 | 作者：小夏 拍板

---

## 一、系统概述

### 1.1 设计目标

万界仙途现有 PvE 玩法是**地图历练**（无限刷）+ **秘境组队**（多人协作）。两者都是"重复消耗型"内容，玩家缺乏一个**大乘后的单人极限挑战目标**。通天塔通过"100 层阶梯式 Boss 战 + 进度继承 + 每日失败配额"，提供：

- **大乘后的目标性内容**——第 1 层就是 T7 ×2.0+ 战力，进塔即考验
- 每日 15 分钟的高密度挑战体验（爬塔 → 卡关 → 思考阵容 → 突破）
- 可视化的成长尺标（"我已经爬到第 X 层"）
- BD（Build）深度的检验场（每层固定怪、固定机制，反复磨同一层逼玩家精进配装与功法）
- **第 51-100 层超出主图战力**，给毕业玩家"无尽之塔"般的极限博弈空间

### 1.2 核心定位

| 维度 | 定位 |
|------|------|
| 玩法类型 | 单人 PvE 极限挑战 |
| 内容载体 | 100 层固定塔（每层独立战斗） |
| 进入门槛 | **大乘（realm_tier ≥ 7）+ Lv 140** |
| 与历练的差异 | 历练是开放式刷图换资源；通天塔是关卡式闯关换进度 |
| 与秘境组队的差异 | 秘境是协作 + 波次 + 装备掉落；通天塔是单人 + 每层独立 + 进度奖励 |
| 时间消耗 | 每天 1-3 次挑战，每次 2-5 分钟（含每层战斗 + 上塔节奏） |
| 核心吸引力 | 进度刻度感、特色 Boss 机制、超主图极限层（奖励待后续设计） |

### 1.3 与已有系统的关系

```
日常资源循环
  ├── 地图历练         → 灵石/经验/装备（无限刷）
  ├── 离线挂机         → 灵石/经验（被动）
  ├── 宗门 Boss        → 周常协作
  └── 秘境组队         → 协作 + 高品质装备

长线挑战目标
  └── 通天塔 ★         → 单人爬塔 + 进度奖励 + BD 检验
```

通天塔**不掉装备、也不引入专属货币**（避免增加经济系统复杂度）。**MVP 阶段暂不设奖励**——先把"爬塔 + 每层机制 + 进度继承 + 失败配额"的核心循环跑通验证。奖励数值与产出渠道在核心玩法验证后再单独迭代设计。

## 二、核心机制

### 2.1 每日次数机制

```
每日失败配额：3 次
每日 00:00（与现有 daily-reset cron 同步）重置
```

**关键细节：**

- **只统计失败**，挑战成功不消耗任何配额 → 鼓励玩家"打通就一直往上爬"
- 当天失败 3 次后，**今日不能再挑战**（不论是哪一层失败）
- 失败次数**全塔通用**（不是每层各自 3 次）
- 主动放弃战斗（中途退出）**计为失败**，防止玩家"看见要输就关页面规避"

**为什么是 3 次而不是更多？**
- 1 次太苛刻，不合放置游戏的轻松基调
- 5+ 次容易让玩家"硬磨"卡关层，丢失"明天再来"的期待
- 3 次刚好够试错，又能制造合理停顿

### 2.2 进度继承机制

```
每个角色在 characters 表上记录：tower_max_floor（已通关的最大层数，初始 0）

挑战入口规则：
  - 默认从 (tower_max_floor + 1) 层开始挑战
  - 玩家也可"自由选层"重温任意已通过的层（MVP 阶段纯练习用；后续接入奖励后用于扫荡）
  - 当前在挑战中的层若失败，下次进入仍从 (tower_max_floor + 1) 开始
```

**为什么允许"自由选层"？**
- 已通关的层不会消耗失败次数（数值上玩家肯定能赢），但能稳定产出每日积分
- 给玩家"卡关时回去扫一扫低层凑积分"的退路

### 2.3 每层重置机制（核心差异点）

每进入一层新战斗，**强制初始化角色战斗状态**：

| 项 | 重置规则 |
|----|---------|
| 当前血量 | 重置为最大血量（满血进场） |
| 主修 / 神通 CD | 全部归零（可立即释放） |
| 战斗 buff / debuff | 全部清空 |
| 不灭金身的"已触发"标记 | 清零（每层都可触发一次） |
| 套装战斗内叠层（十三枪、刀狂等） | 清零（自然由"新战斗"重置） |
| 当前回合数 | 归零 |

**设计意图：**
- 玩家每层都是"满状态进场"，不会因为上一层惨胜导致下一层无法打
- 这让"卡关"卡的是**配装/BD 深度**而非"上一层我血量已经被磨光"
- 让"通天塔每层 = 一个独立的 Boss 战谜题"，鼓励玩家针对特定机制重新搭配功法

**与地图历练对比：**
- 地图历练：多波次连续战斗，血量不重置、CD 不重置 → 检验**续航**
- 通天塔：每层独立、满状态进入 → 检验**爆发与机制应对**

### 2.4 战斗结束规则（单层）

| 情况 | 结果 |
|------|------|
| 击败本层全部敌人 | 成功，**默认自动进入下一层挑战**（玩家可手动按"暂停下塔"中止） |
| 玩家被击杀 | 失败，扣 1 次失败配额；最大层数不刷新；自动停下，等待玩家选择"再战"或"下塔" |
| 回合数达上限（每层 100 回合） | 失败，扣 1 次失败配额；同上 |
| 玩家中途退出 | 视为失败，扣 1 次失败配额 |

> 单层 100 回合上限是为了防止"龟缩流"无限拖延。普通配装 10-30 回合即可清场（中后期 Boss 25-35 回合），100 回合给 DOT/续航/反伤等长线 BD 留出充足发挥窗口。

### 2.5 连续挑战流程

```
玩家点击【挑战】
  ↓
进入第 N 层（N = 最高层+1 或自选层）
  ↓
[战斗] → 重置血量+CD → 服务端一次性算完战斗 → 返回日志
  ↓
胜利？
  ├── 是：tower_max_floor = max(tower_max_floor, N)
  │       展示战斗结算（含简短播放；MVP 阶段无奖励，纯展示战斗数据）
  │       ↓
  │       【默认行为】结算页倒计时 3 秒后自动进入第 N+1 层
  │           ├── 玩家不操作 → 直接连战下一层
  │           └── 玩家点击 [暂停下塔] → 中止连战，退出塔
  │       N 自增，回到顶部
  └── 否：daily_fail_count += 1
          展示战斗日志 + "今日剩余尝试 X 次"
          ↓
          自动停下（不连战），玩家手动选择：
          ├── 剩余 = 0 → 仅显示 [下塔]
          └── 剩余 > 0 → 显示 [再战 第 N 层] / [下塔]
```

**默认连战是核心爽点**：胜利后无需任何操作即可一路上塔，玩家只需在想停时主动按"暂停"；失败后则自动停下，避免连续失败用光当日配额。

## 三、塔层设计

### 3.1 100 层难度节奏（极限挑战定位）

通天塔**不对应玩家境界曲线**，而是从"大乘后的 BD 检验场"一路爬到"远超主图的无尽极限"。第 1 层就用主图 T7 大乘 ×2.0+ 战力起步——意味着只有打通主图大乘内容的玩家才有资格踏入塔门。

| 层段 | 对位主图 tier | 怪物 power 区间 | 倍率 | 体验定位 |
|------|---------|---------------|---------|---------|
| 1-15 | **T7 大乘** | 60k ~ 235k | T7 ×0.75-2.0 | 入塔门槛，1-3 trait 堆叠 |
| 16-25 | **T8 飞升初** | 225k ~ 600k | T8 ×1.0-1.5 | 紫红装毕业 + 套装协同 |
| 26-35 | **T9 飞升中** | 250k ~ 870k | T9 ×1.0-1.3 | BD 深度检验段 |
| 36-50 | **T10 飞升后** | 800k ~ 2.95M | T10 ×1.5-2.5 | 神器流派出场 |
| 51-65 | **T11 混元初** | 3M ~ 8.5M | T11 ×1.3-1.6 | 混元曲线初验 |
| 66-80 | **T12 混元中** | 10M ~ 38M | T12 ×1.3-1.6 | 混元末期玩家挑战 |
| 81-90 | **T13 道域** | 35M ~ 90M | T13 ×1.3-1.5 | 法则装备验证 |
| 91-95 | **T14 法则** | 120M ~ 330M | T14 ×1.0-1.5 | 顶级玩家差距拉开 |
| 96-100 | **T15 终焉** | 350M ~ 1B | T15 ×0.7-1.5 | 塔主"镇塔之灵" 1B power |

> **设计原则：** 通天塔是大乘后才有意义的内容。它的奖励（修为/万能残页/突破丹/太古精魂/道果结晶）都是中后期-毕业玩家才用得上的稀有材料，与"低境界教学"完全不矛盾。
> **倍率取舍（v3 削弱版）：** 因每层重置满血 + CD 满，玩家爆发比主图刷怪更猛，所以塔比主图同 tier 高 1.0-2.5 倍即可。低于 ×1 的段位（如 T7/T15）通过 trait 复合补足挑战感。
> **段位映射主图：** 每段直接对应主图已存在的 tier（T7-T15），drop_table 用 `tower_t7` ~ `tower_t15`，引擎已支持。
> **塔主"镇塔之灵"：** 第 100 层 power 1B，对位 T15 终焉道祖（700M）的 1.43 倍，6 trait 全机制集大成。

### 3.2 关键里程碑层

每 5 层设置一个"层主"，每 25 层设置一个"中 Boss"，让爬塔节奏不平淡：

| 层数 | 类型 | 特征 |
|------|------|------|
| 每 5/10/15/...（每 5 层尾） | **层主** | 1 怪，机制更刁钻，power 比同段普通层 ×1.3 |
| 第 25 / 50 / 75 层 | **中 Boss** | 1 怪，三阶段变身（B01）；power 比同段普通层 ×1.8 |
| 第 100 层 | **塔主** | "镇塔之灵" — 三阶段变身 + 五行轮转 + 全机制集大成；power ×3.0 |

> 大乘起步后，每 5 层就是一道坎，玩家需要持续磨配装。这也意味着 100 层一共有 ~20 个层主 + 4 个中 Boss + 1 个塔主 = **25 个有辨识度的关卡**。

### 3.3 每层结构

每层固定 **1 或 2 个怪物**，配置全服一致（**所有玩家打同一层看到的是相同的怪物模板**）：

| 层数 | 怪物数量 | 备注 |
|------|---------|------|
| 1-25 | 1 怪 | 每层只有一个特色怪 |
| 26-70 | 2 怪 | 主怪 + 副怪（副怪通常是辅助/控制类） |
| 71-100 | 1-2 怪 | 中 Boss / 塔主层为 1 怪（避免战斗过长），其他为 2 怪 |

> **不出现波次**（与秘境的多波次区分），让每层"短而精"。

### 3.4 怪物模板复用现有引擎

通天塔怪物**完全复用** `server/engine/battleEngine.ts` 的 `MonsterTemplate` 接口（与历练、秘境同源）：

```typescript
interface MonsterTemplate {
  name: string
  power: number          // 战力，引擎据此推算 HP/ATK/DEF/SPD
  element: string | null
  role: 'balanced' | 'dps' | 'tank' | 'speed' | 'boss'
  exp: number
  stone_min: number
  stone_max: number
  drop_table: string     // 通天塔不掉装备，此字段写 'tower_no_drop'
  // 通天塔特有扩展（新增字段，可选）：
  towerTraits?: TowerTrait[]   // 见 §四 特色机制
}
```

## 四、特色机制库（怪物 Trait）

每层怪物从下面 24 类机制中挑选 1-3 个组合，构成"这层的解法谜题"。

### 4.1 机制分类总览

| 分类 | 机制 | 典型层数 |
|------|------|---------|
| **狂暴类** | 狂暴、滚雪球、爆发 | 中后期 |
| **续航类** | 嗜血、再生、复活 | 全段 |
| **防御类** | 减伤、护盾叠加、圣盾期 | 中段 |
| **元素类** | 元素免疫、元素吸收、元素反制 | 中后段 |
| **反制类** | 反伤、镜像、反击 | 中段 |
| **控制类** | 群冻、群眩、沉默、束缚 | 全段 |
| **DOT 类** | 命中附毒、命中附烧、命中附流血 | 入门段 |
| **特殊类** | 召唤、分身、阶段变身、真伤、先手 | Boss 层 |

### 4.2 详细机制定义

| ID | 机制名 | 触发与效果 | 设计意图 |
|----|--------|-----------|---------|
| T01 | 狂暴 | 战斗 5 回合后，攻击 +50%，速度 +30% | 检验玩家爆发速度 |
| T02 | 滚雪球 | 每次造成伤害后，自身攻击 +5%（最多叠 10 层） | 鼓励早断输出 |
| T03 | 爆发 | 会心率 +50%、会心伤害 +50% | 检验防御与闪避 |
| T04 | 嗜血 | 攻击造成伤害的 30% 转为自身回血 | 检验单击爆发是否能压住回血 |
| T05 | 再生 | 每回合恢复最大血量 5% | 玩家 DPS 必须高于 5%/回合 |
| T06 | 复活 | 血量首次降至 0 时，恢复 50% 血量并重置自身 buff（仅触发 1 次） | 检验"持续输出能力" |
| T07 | 减伤 | 受到的所有伤害减少 30% | 检验破甲 / 真伤词条 |
| T08 | 护盾叠加 | 每回合开始为自己生成 maxHp 5% 的护盾（可叠加，上限 30%） | 引导带护盾穿透/打断流派 |
| T09 | 圣盾期 | 战斗前 3 回合免疫一切伤害（仍可被控制） | 检验控制链 + 后期续航 |
| T10 | 元素免疫 | 对自身同元素伤害免疫；同时免疫该元素的所有 debuff | 检验玩家是否被锁五行 |
| T11 | 元素吸收 | 受到自身同元素伤害时，转化为自身回血 100% | 强逼玩家避开错误元素 |

| T13 | 反伤 | 受到伤害的 25% 反弹给攻击者（受伤反烧/反毒规则同主战引擎） | 检验是否带反伤抵抗 |
| T14 | 镜像 | 每 3 回合开始时，将上一回合受到的总伤害的 100% 反弹一次 | 鼓励玩家分散输出节奏 |
| T15 | 反击 | 受到普通攻击时，立即反击一次（伤害 = 自身攻击 ×80%） | 让"高频普攻"流派吃亏 |
| T16 | 群冻附带 | 攻击命中时 30% 概率冻结 1 回合 | 检验抗控/免疫词条 |
| T17 | 群眩附带 | 攻击命中时 25% 概率眩晕 1 回合 | 同上 |
| T18 | 沉默 | 攻击命中时 30% 概率沉默 2 回合（无法释放神通） | 检验玩家是否过度依赖某一神通 |
| T19 | 束缚 | 攻击命中时 30% 概率束缚 2 回合（无法闪避） | 检验闪避流派 |
| T20 | 命中附毒 | 攻击命中时叠 1 层中毒（最大叠至 5 层） | 入门 DOT 教学层 |
| T21 | 命中附烧 | 攻击命中时叠 1 层灼烧（最大叠至 5 层） | 同上 |

| T24 | 先手 | 战斗开始时强制先手一回合（不论速度） | Boss 专用，制造紧迫感 |

## 五、奖励系统

> **MVP 阶段：仅发放"一次性"奖励**——称号 + 永久属性加成。
> "循环型"奖励（灵石/经验/万能残页等可重复刷的物品）暂不发放，等核心玩法上线、玩家行为数据回收后再针对性设计。

### 5.1 奖励分类与发放策略

| 类别 | MVP 即发？ | 说明 |
|------|:---------:|------|
| 称号 | ✅ | 纯展示性，不影响平衡 |
| 一次性永久属性加成 | ✅ | 仅首次通关触发，不可重复刷；属于成就锚点 |
| 头像框 | ✅（Phase 3 第 100 层） | 复用现有头像框系统 |
| 灵石 / 经验 / 万能残页 / 突破丹 / 太古精魂 / 道果结晶 | ❌（Phase 4） | 循环型物品，可重复扫荡产出，需经济平衡校准 |
| 装备 | ❌（永不） | 装备生态留给秘境与历练 |
| 专属货币 | ❌（永不） | 不增加经济复杂度 |

### 5.2 称号清单（MVP 即上线）

每个称号在玩家**首次通关对应层**时自动授予，写入现有 `character_titles` 表（复用宗门/秘境的称号系统）：

| 层数 | 称号 | 颜色档次（建议） | 描述 |
|------|------|----------------|------|
| 第 15 层（大乘段终点层主） | **塔下行者** | 蓝（普通） | 踏过通天塔大乘段的修士 |
| 第 25 层（第一个中 Boss） | **塔中过客** | 紫（稀有） | 过半见证通天塔奥秘 |
| 第 50 层（第二个中 Boss） | **半塔之主** | 金（史诗） | 站在通天塔半山腰俯瞰众生 |
| 第 75 层（第三个中 Boss，Phase 3） | **塔顶遥望** | 金（史诗） | 已能远眺塔尖，触手可及 |
| 第 100 层（塔主"镇塔之灵"，Phase 3） | **通天塔主** | 红（传说） | 通天塔登顶者，全服稀有称号 |

> 称号系统**不互相覆盖**：玩家会同时拥有"塔下行者"、"塔中过客"……可在角色档案/聊天前缀里自由切换显示。
> 颜色档次仅作建议，与现有称号 UI 复用。

### 5.3 一次性永久属性加成（MVP 即上线）

每个里程碑层**首次通关**触发，永久写入角色面板，**不可重复获得**：

| 层数 | 永久奖励 | 累计 | 备注 |
|------|---------|-----:|------|
| 第 25 层（中 Boss） | 全属性 +1% | +1% | 第一个里程碑 |
| 第 50 层（中 Boss） | 全属性 +2% | +3% | 半塔之主，跃迁式提升 |
| 第 75 层（中 Boss，Phase 3） | 全属性 +3% | +6% | 进入终极阶段 |
| 第 100 层（塔主，Phase 3） | 全属性 +5% | +11% | 通天塔主，全塔通关上限 |

**全属性加成**说明：
- 同时影响 **气血 / 攻击 / 防御 / 速度** 四项基础属性的百分比
- 100 层全部首通 = 永久 +11% 全属性，相当于"半套金级被动"的强度
- 与道果结晶（现有系统的永久属性道具）**叠加生效**，不冲突
- 一次性触发后写入 `characters.permanent_stats_*` 字段（复用现有道果结晶机制），战斗引擎自动读取

**为什么数值这么克制（11% 上限）？**
- 大乘 / 飞升玩家基础属性已数十万级别，11% 加成约 = 一件红装一条副属性的量级
- 如果数值过高，会让"没爬过通天塔"的玩家在 PvP / 宗门战中天然劣势 → 强迫机制
- 11% 既能让爬塔玩家自豪，又不形成"必刷"压力

### 5.4 物品奖励待定项（Phase 4 启动）

后续物品奖励设计需要回答的问题：

- [ ] 首通奖励梯度（按层段 1-15 / 16-30 / ... 设置）
- [ ] 层主层、中 Boss 层、塔主层的加倍倍率
- [ ] 每日扫荡的具体数值公式
- [ ] 头像框样式（与第 100 层称号联动）
- [ ] 与成就系统的循环型奖励联动（参见 §10.3）

> **MVP 上线时**：完整跑通战斗 + 进度记录 + 失败配额 + **称号自动授予 + 一次性永久属性加成**；战斗结算页循环型物品奖励列表显示"敬请期待"占位。玩家通过"已通最高层 + 解锁称号 + 永久属性"获得成就感与持续动力，足以验证"爬塔本身是否有趣"。

### 5.5 数据结构预留

| 项 | MVP 实现 | Phase 4 补完 |
|----|--------|------------|
| 称号 | 写入 `character_titles` 表 | — |
| 一次性永久属性 | 复用现有 `characters.permanent_*_pct` 字段（道果结晶同源） | — |
| 循环型物品奖励 | 数据库与 API 字段已预留（`tower_battles.rewards` 等） | 仅需填入数值 |

## 七、数据库设计

### 7.1 新增表与字段

```sql
-- ==================== 通天塔系统 ====================

-- 1. 角色塔进度（直接挂在 characters 表上，访问最频繁）
ALTER TABLE characters ADD COLUMN IF NOT EXISTS tower_max_floor          SMALLINT NOT NULL DEFAULT 0;
ALTER TABLE characters ADD COLUMN IF NOT EXISTS tower_daily_fail         SMALLINT NOT NULL DEFAULT 0;
ALTER TABLE characters ADD COLUMN IF NOT EXISTS tower_daily_date      DATE;
ALTER TABLE characters ADD COLUMN IF NOT EXISTS tower_last_sweep_date DATE;   -- 每日扫荡最近领取日期

-- 2. 塔战斗记录（保留最近 N 次，用于查询日志/复盘）
CREATE TABLE IF NOT EXISTS tower_battles (
    id              SERIAL PRIMARY KEY,
    character_id    INTEGER NOT NULL REFERENCES characters(id),
    floor           SMALLINT NOT NULL,                -- 挑战的层数
    result          VARCHAR(10) NOT NULL,              -- victory / defeat
    total_turns     INTEGER NOT NULL DEFAULT 0,
    damage_dealt    BIGINT NOT NULL DEFAULT 0,
    damage_taken    BIGINT NOT NULL DEFAULT 0,
    battle_log      JSONB,                             -- 完整战斗日志
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_tower_battles_char ON tower_battles(character_id, created_at DESC);

-- 3. 首通记录（用于幂等判定 + 历史浏览）
CREATE TABLE IF NOT EXISTS tower_clears (
    id              SERIAL PRIMARY KEY,
    character_id    INTEGER NOT NULL REFERENCES characters(id),
    floor           SMALLINT NOT NULL,
    cleared_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    battle_id       INTEGER REFERENCES tower_battles(id),
    UNIQUE(character_id, floor)                        -- 每层每角色仅一条
);

CREATE INDEX idx_tower_clears_char ON tower_clears(character_id, floor);
```

> 不引入专属货币 → 不需要"商店购买记录"表。所有奖励直接 `UPDATE characters SET spirit_stone = ...` 或调用现有 inventory 接口入库。

### 7.2 与 daily-reset 的集成

复用现有 `server/api/cron/daily-reset.post.ts`，新增一行：

```sql
-- 通天塔每日失败次数清零（懒重置：实际重置在玩家访问时按日期判断也可，cron 仅作兜底）
UPDATE characters SET tower_daily_fail = 0, tower_daily_date = CURRENT_DATE
WHERE tower_daily_date IS NULL OR tower_daily_date < CURRENT_DATE;
```

> 实现上推荐**懒重置**：每次玩家访问 `/api/tower/info` 或 `/api/tower/challenge` 时，先比对 `tower_daily_date`，与今日不同则清零并更新日期。这样跨时区或 cron 失败时也不会出错。

### 8.1 接口总览

| 方法 | 路径 | 说明 |
|------|------|------|

### 8.2 关键接口详细设计

#### GET /api/tower/info

```typescript
// Response
{
  success: true,
  info: {
    max_floor: 47,                  // 已通关最高层
    next_floor: 48,                 // 下次默认挑战层
    daily_fail_used: 1,             // 今日已失败次数
    daily_fail_max: 3,              // 今日最大失败次数
    can_challenge: true,            // 今日是否还能挑战
    can_sweep: true,                // 今日是否还能领扫荡
    sweep_reward: {                 // 扫荡可得（按当前 max_floor 计算）
      spirit_stone: 10000,
      exp: 15000,
      universal_page: 1             // 仅 max_floor ≥ 30 才有
    },
    total_floors: 100               // 塔总层数
  }
}
```

#### GET /api/tower/floor/:n

```typescript
// Response
{
  success: true,
  floor: {
    n: 48,
    name: "灵狐之惑",                // 层名（可选，纯氛围）
    monsters: [
      {
        name: "九尾灵狐",
        element: "fire",
        role: "dps",
        power: 18000,
        traits: [
          { id: "T01", name: "狂暴", desc: "5 回合后攻击 +50%、速度 +30%" },
          { id: "T17", name: "群眩附带", desc: "攻击命中时 25% 概率眩晕 1 回合" }
        ]
      }
    ],
    is_layer_lord: false,            // 是否层主
    is_boss: false,                  // 是否中 Boss

      exp: 50000,
      universal_page: 2

```

> 让玩家**进战前能看到怪物机制**，鼓励"读机制 → 调配装/功法 → 再战"。

#### POST /api/tower/challenge

```typescript
// Request
{ floor: 48 }                        // 目标层（必须 ≤ max_floor + 1）

// Response
{
  success: true,
  battle: {
    id: 1234,
    floor: 48,
    result: "victory" | "defeat",
    total_turns: 24,
    log: [...],                      // 完整战斗日志
    rewards: {                       // 仅胜利时返回
      is_first_clear: true,          // 是否本次为首通
      spirit_stone: 30000,
      exp_gained: 50000,
      extra_drops: [{ type: 'item', id: 'universal_page', quantity: 2 }]
    }
  },
  state_after: {
    max_floor: 48,                   // 更新后的最高层
    daily_fail_used: 1,
    can_challenge: true
  }
}
```

**服务端流程：**

1. 校验：登录、不在挂机、不在其他战斗
2. 校验：`floor ≤ max_floor + 1`（不能跳层挑战）
3. 校验：`daily_fail_used < daily_fail_max`（**只在挑战未通过的层时校验**；扫荡或重温已通过的层不消耗失败配额，但**也不算失败次数**——保险起见，仍允许玩家自由挑战已通过的层；失败时若是已通过的层，**不扣失败次数**）
   - 简化版：失败次数仅在挑战 `> max_floor` 的层时才计入失败次数。重温已通过层失败不计。
4. 加载角色数据 + 装备 + 功法
5. 加载本层怪物配置（含 traits）
6. 执行战斗（复用 battleEngine，扩展 trait 应用钩子）
7. 入库：tower_battles + 必要时 tower_clears + 更新 characters
8. 返回结果

> 注意：每场战斗都是**一次性原子计算**（与现有 /api/battle/fight 一致），无中间状态。

#### POST /api/tower/sweep

```typescript
// Response
{
  success: true,
  reward: {
    spirit_stone: 10000,
    exp: 15000,
    universal_page: 1               // 当 max_floor ≥ 30 时返回
  }
}
```

服务端校验 `tower_last_sweep_date < CURRENT_DATE`，否则返回"今日已领取"。

### 9.5 状态管理（stores/tower.ts）

```typescript
interface TowerState {
```

```typescript
})
```

## 十二、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 100 层 traits 配置工作量大 | 上线慢 | Phase 1 先做 1-25 层（大乘~飞升初段），后续分期补完 |
| 玩家卡关焦虑 | 留存掉 | MVP 无奖励压力 + 不掉装备减少 FOMO + 失败 3 次硬限轻量化；后续上奖励时再加每日扫荡作为兜底 |
| 失败 3 次后玩家流失 | 当日活跃 | 通天塔不是每日必做，玩家失败后转去刷图/秘境，反而分流体验 |
| 进度差大的玩家间排行无意义 | 排行榜冷清 | 排行榜以"最高层"为主，配合"达成时间"二级排序，让冲层选手有看头 |
| trait 机制过于花哨破坏战斗平衡 | 玩家觉得不公平 | 每个 trait 必须有明确的"反制方法"（如 T07 减伤 → 破甲；T11 元素吸收 → 换元素），战前预览页明示机制 |
| 100 层塔主过强一直无人通 | 长尾内容浪费 | 第 100 层奖励够厚（永久 +5% 全属性），全服首通可设全服公告制造话题 |
| 与秘境争抢玩家时间 | 重叠 | 每日定位不同：秘境 = 团队晚上 30 分钟；通天塔 = 单人 5-10 分钟碎片时间 |
| 进度继承导致玩家偷懒 | 体验单调 | 每层固定模板让玩家"打过即过"；新内容压力转移到塔主层与赛季 |

### A.4 数值合理性自检（粗算）

> 玩家假设按段位毕业（每段都是对位主图刚毕业的玩家），所以 TTK 一致——这是塔的核心设计：每段考验 BD 而非数值差。

| 层 | Boss Power | Tier | 玩家假设阶段 | 玩家裸打 TTK | 备注 |
|---:|----------:|-----:|------------|-------------:|------|
| 25 | 600k | T8 | 飞升初毕业 | 22-30 回合 | 中 Boss，三阶段 |
| 50 | 2.95M | T10 | 飞升后毕业 | 25-35 回合 | 中 Boss，三阶段 |
| 75 | 32M | T12 | 混元中毕业 | 30-40 回合 | 中 Boss，三阶段+五行轮转 |
| 100 | 1B | T15 | 终焉毕业 | 40-60 回合 | 塔主，6 trait 全机制集大成 |

> 因每层重置满血 + CD 满，玩家爆发明显比主图刷怪强；上述 TTK 估算已纳入"塔比主图 ×1-2.5"的倍率。
> 100 回合上限对 TTK 60 的塔主仍有 40 回合余量，可接受。
> 玩家若装备未齐毕业 / 元素配置不匹配，TTK 翻倍很正常——塔主关需要"反复磨"是设计意图。

## 附录 C：与秘境组队的对位表

| 维度 | 秘境组队 | 通天塔 |
|------|---------|-------|
| 玩法 | 多人协作 | 单人挑战 |
| 内容载体 | 6 个秘境 × 3 难度，多波次 | 100 层固定塔，每层独立 |
| 进度模型 | 每次刷一遍换奖励 | 阶梯式累积，进度继承 |
| 时间成本 | 每场 2-5 分钟，组队等待+战斗 | 每层 1-3 分钟，连续作战 |
| 失败成本 | 30% 奖励 | 扣 1 次失败配额 |
| 核心产出 | 装备（紫/金/红） | _MVP 阶段无奖励_；后续设计为现有道具（灵石/经验/万能残页/突破丹/太古精魂/道果结晶） |
| 专属货币 | 秘境积分 | **无**（直接发现有道具） |
| 商店重点 | 装备宝箱 + 功法卷 + 升级材料 | 无商店 |
| 检验维度 | 团队配合、阵容多元化 | 单角色 BD 深度 |
| 社交属性 | 高（公开大厅 + 同宗门加成） | 低（仅排行榜） |
| 每日次数 | 2-5 次（按境界） | 失败 3 次锁，胜利无限 |

> 两者形成互补：**秘境给装备，通天塔给修为/升级材料**；**秘境靠人，通天塔靠己**。
> 通天塔不引入新货币和商店，把"修为/升级材料"的产出渠道直接挂在爬塔进度上，简化经济。
